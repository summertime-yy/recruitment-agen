## 一、问题陈述

### 1.1 当前痛点

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **工具切换频繁** | HR需要在项目管理平台查看任务包 → 切换到BOSS/猎聘发布职位 → 切换回邮箱收取简历 → 使用Excel/纸质记录筛选结果 | 效率低、易出错、上下文丢失 |
| **数据不互通** | 任务包在项目管理平台，招聘在BOSS/邮箱，候选人数据在Excel | 无法自动关联、无法追溯、重复录入 |
| **信息孤岛** | 无法从一个入口访问：邮箱、项目管理系统、人事系统、企微 | 协作成本高、信息延迟 |
| **移动端需求低** | 用户反馈"移动端支持不重要，主要在桌面端用" | 应聚焦桌面端企微体验 |

### 1.2 用户原声（5次访谈摘录）

> "切换工具太麻烦，希望在一个地方完成所有工作"
> "希望能直接链接到我们产品中的其他对象（邮箱、项目管理系统、人事系统、企微）"
> "移动端支持不重要，主要在桌面端用"
> "希望AI能帮我自动写JD，现在每次都要从零开始写"
> "收到简历后，希望AI能自动筛选打分，而不是我一个个看"

### 1.3 竞品速览

| 产品 | 定位 | 差距/机会 |
|------|------|----------|
| **Moka AI** | ATS系统内置AI筛选，Web端操作 | 非企微原生，HR仍需切换到Web |
| **北森AI招聘** | 一体化HR SaaS，功能全面 | 重、贵、需要系统迁移，不适合轻量场景 |
| **海纳AI** | AI简历解析+人岗匹配 | 独立工具，不集成企微生态 |
| **我们的机会** | 企微原生、@机器人交互、轻量零切换 | 填补"企微内完成招聘流程"的空白 |

## 二、目标与非目标

### 2.1 产品目标

| 目标 | 说明 | 衡量方式 |
|------|------|---------|
| **统一入口** | HR通过企微@机器人完成JD生成和简历筛选 | 企微交互成功率 > 90% |
| **智能生成** | 根据任务描述自动生成标准化JD | JD生成时间 < 2分钟 |
| **智能筛选** | AI自动解析简历、多维度评分、推送筛选报告 | 筛选Top3与HR盲评重叠率 ≥ 70% |
| **减少切换** | 从"JD生成+简历筛选"环节消除工具切换 | 该环节工具切换数从4→1 |

### 2.2 产品定位声明

> **智能招聘智能体是HR的AI辅助工具，不是HR的替代品。** 所有AI输出均为建议，最终决策权始终在HR手中。

### 2.3 非目标（明确不做）

- ❌ **移动端APP**：用户明确反馈桌面端为主，不开发独立APP
- ❌ **替代HR判断**：AI输出均为建议，最终决策权始终在HR
- ❌ **BOSS/猎聘自动发布**：当前无公开API，不做自动发布（仅生成JD供HR手动发布）
- ❌ **全自动无人工审核**：关键决策必须有人工确认环节
- ❌ **薪酬谈判**：仅做筛选和初步沟通，薪酬谈判由HR人工完成

## 三、用户故事

### 3.1 核心用户故事（MVP）

#### US-1：根据任务描述生成岗位JD

> 作为HR人力专员，我想要在企微群中@招聘机器人并输入任务包描述，以便于自动生成一份结构化的岗位JD。

**验收标准：**

- **Scenario 1 — 正常生成JD**
  - Given HR在企微群中发送 `@招聘助手 帮我招一个嵌入式软件工程师，负责芯片验证和驱动的开发，需要熟悉UVM和SystemVerilog`
  - When 系统识别意图为"生成JD"
  - Then 60秒内返回一条包含以下内容的Markdown消息：
    - 岗位名称
    - 岗位职责（至少3条）
    - 任职要求（硬性条件+加分项各至少2条）
    - 工作地点、汇报关系
  - And 消息底部附带操作提示：`💡 回复"修改"可提出修改意见` `💡 回复"确认"保存JD`

- **Scenario 2 — 描述信息不足**
  - Given HR发送 `@招聘助手 帮我招个人`
  - When 系统判断信息不足以生成JD
  - Then 30秒内返回追问消息，至少询问：岗位名称、核心技能、团队背景
  - And 不输出不完整的JD

- **Scenario 3 — 修改JD**
  - Given HR已收到一份生成的JD
  - When HR回复 `修改：技能要求改成必须熟悉Python和C++`
  - Then 系统在原JD基础上更新指定字段，重新推送修改后的JD
  - And 保留其他未涉及的字段不变

---

#### US-2：手动提交简历并解析

> 作为HR人力专员，我想要把候选人的简历直接发给招聘助手，以便于AI自动解析出结构化信息。

**验收标准：**

- **Scenario 1 — 上传PDF简历文件**
  - Given HR在企微群中将一份PDF简历文件拖入输入框并发送
  - When 系统接收到PDF文件
  - Then 60秒内解析完成，推送包含以下字段的结构化信息：
    - 姓名、联系方式（脱敏显示）
    - 最高学历、毕业院校、专业
    - 工作年限、最近3段工作经历（公司/职位/时长）
    - 技能标签列表
  - And 消息底部提示：`✅ 简历解析完成，已关联到当前岗位[嵌入式软件工程师]`

- **Scenario 2 — 粘贴简历文本**
  - Given HR直接粘贴简历纯文本内容发送给招聘助手
  - When 系统识别为简历内容
  - Then 同Scenario 1完成解析

- **Scenario 3 — 图片简历（OCR兜底）**
  - Given HR发送一张图片格式的简历
  - When 系统识别为图片
  - Then 先调用OCR识别 → 再进行结构化解析
  - And 如OCR置信度 < 80%，附提示：`⚠️ 图片识别可能有偏差，建议用PDF或文本格式`

- **Scenario 4 — 非简历文件**
  - Given HR误发了一个非简历文件
  - When 系统无法从中提取候选人的结构化信息
  - Then 返回提示：`⚠️ 未识别到简历信息，请确认文件内容。支持的格式：PDF、Word、图片、纯文本`

---

#### US-3：AI筛选评分并推送报告

> 作为HR人力专员，我想要招聘助手对收集到的简历进行自动评分，以便于快速锁定优质候选人。

**验收标准：**

- **Scenario 1 — 触发筛选**
  - Given 当前岗位已关联至少3份解析完成的简历
  - When HR发送 `@招聘助手 开始筛选` 或 `@招聘助手 看看这批简历`
  - Then 系统对所有关联简历进行多维度评分
  - And 90秒内推送筛选报告，包含：
    - 汇总：总简历数、筛选完成数、推荐面试数
    - Top 3 候选人列表（姓名、总分、各维度得分）
    - 每位候选人匹配亮点一句话总结

- **Scenario 2 — 评分维度**
  - Given 系统对简历进行评分
  - Then 评分包含以下5个维度（各0-20分，满分100）：
    - 技能匹配度（JD要求的技能 vs 候选人技能）
    - 经验匹配度（工作年限、行业背景 vs JD要求）
    - 学历匹配度（学历层次、专业相关性）
    - 项目匹配度（项目复杂度、角色 vs JD要求）
    - 稳定性（平均在职时长、跳槽频率）

- **Scenario 3 — 简历数量不足**
  - Given 当前岗位关联简历数 < 3
  - When HR触发筛选
  - Then 返回提示：`📭 当前仅有X份简历，建议至少积累3份后再筛选。是否想预览已解析的简历？`

- **Scenario 4 — 查看单份简历详情**
  - Given 筛选报告已推送
  - When HR回复 `查看 张三 的详细评分`
  - Then 推送该候选人的完整评分明细：每个维度的得分理由、匹配的关键词、不匹配的差距分析

---

#### US-4：查看岗位进度和已收集简历

> 作为HR人力专员，我想要随时查看某个岗位的招聘进度，以便于了解当前已收集的简历和筛选状态。

**验收标准：**

- **Scenario 1 — 查询岗位进度**
  - Given HR发送 `@招聘助手 看下进度` 或 `@招聘助手 招聘情况`
  - When 系统识别当前群关联的活跃岗位
  - Then 推送进度摘要：
    - 岗位名称 + 状态（JD已生成/筛选中/已完成）
    - 已收集简历数、已解析简历数、已评分简历数
    - 最近一次操作时间和操作人

- **Scenario 2 — 无活跃岗位**
  - Given 当前群没有关联的岗位
  - When HR查询进度
  - Then 返回引导：`📋 暂无活跃岗位。试试 @招聘助手 帮我招一个人 开始新的招聘流程`

## 四、需求说明

### 4.1 MVP需求（1-2周发布）

#### 4.1.1 MVP范围一句话总结

> **最小闭环**：企微群中 @招聘机器人 → 输入任务描述 → AI生成JD → HR手动提交简历 → AI解析评分 → 推送筛选报告。

#### 4.1.2 功能需求

| 功能模块 | 功能点 | 优先级 | 说明 |
|---------|-------|:------:|------|
| **企微交互** | @机器人触发 | P0 | 支持在企微群中@机器人，输入自然语言指令 |
| **企微交互** | 意图识别 | P0 | AI自动识别意图：生成JD、提交简历、筛选简历、查询进度 |
| **企微交互** | Markdown消息推送 | P0 | 推送格式化的JD、简历解析结果、筛选报告 |
| **企微交互** | 文件接收（简历） | P0 | 支持接收PDF/Word/图片文件 |
| **JD生成** | 根据任务描述生成JD | P0 | AI分析任务描述，提取技能要求，生成标准化JD |
| **JD生成** | JD修改迭代 | P0 | HR可通过自然语言提出修改意见，AI更新JD |
| **JD生成** | 信息不足追问 | P1 | 当输入信息不足以生成JD时，主动追问关键信息 |
| **简历解析** | 结构化信息提取 | P0 | 从简历中提取姓名/学历/工作经历/技能标签 |
| **简历解析** | 多格式支持 | P1 | 支持PDF、Word、图片（OCR）、纯文本 |
| **简历解析** | 简历关联岗位 | P0 | 解析后自动关联到当前活跃岗位 |
| **简历筛选** | 多维度评分 | P0 | 技能/经验/学历/项目/稳定性五维度评分 |
| **简历筛选** | 筛选报告推送 | P0 | Top候选人列表 + 评分详情 |
| **进度查询** | 岗位进度查看 | P1 | 查看当前岗位的简历收集和筛选状态 |

### 4.2 企微交互规范

#### 4.2.1 支持的指令列表（MVP）

| 指令分类 | 用户表述示例 | 触发行为 |
|---------|-------------|---------|
| **生成JD** | `帮我招一个XX` / `招聘XX` / `需要XX岗位` / `写个JD` | 启动JD生成流程 |
| **修改JD** | `修改JD` / `改成XX` / `加上XX要求` / `去掉XX` | 修改当前JD |
| **确认JD** | `确认` / `OK` / `就这样` / `没问题` | 保存JD，进入简历收集状态 |
| **提交简历** | （直接发送文件或粘贴简历文本） | 触达简历解析 |
| **筛选简历** | `筛选` / `开始筛选` / `看看这批` / `打分` | 触发多维度评分 |
| **查询进度** | `进度` / `怎么样了` / `多少份了` / `情况` | 推送岗位进度 |
| **查看详情** | `查看 张三` / `张三 详情` / `张三 评分` | 推送单份简历评分明细 |

#### 4.2.2 Fallback 策略

当AI无法识别意图时，统一回复：

```
🤔 抱歉，我没理解您的意思。我可以帮您：
• 生成JD — 输入"帮我招一个[岗位]"
• 提交简历 — 直接发送简历文件或粘贴文本
• 筛选简历 — 输入"开始筛选"
• 查询进度 — 输入"进度"
💡 或者直接描述您想做什么，我尽力理解。
```

**工作地点**：上海
**汇报对象**：芯片验证组组长

**硬性条件**

#### 7.2.1 多轮对话状态管理（详细方案）

**问题**：企微群聊是"无状态"消息通道，每次@机器人都是独立事件。HR 可能中断对话、切换话题、或多人交替操作，如何在无状态回调中维持会话上下文？

**方案：Redis会话状态机**

```
存储模型:
  Key:    conv:{corp_id}:{group_id}
  Value:  {
    "state": "COLLECTING",        // 当前状态
    "job_id": "job_20260623_001", // 关联的岗位ID
    "job_title": "嵌入式软件工程师",
    "resume_count": 5,             // 已收集简历数
    "created_at": 1719130000,
    "updated_at": 1719130300
  }
  TTL:    300s (5分钟无操作自动过期回到IDLE)
```

**6个状态 + 转换规则**：

| 状态 | 含义 | 接受的用户操作 | 下一状态 |
|------|------|-------------|---------|
| `IDLE` | 空闲，等待指令 | `@机器人 帮我招一个XX` | → `JD_GENERATING` |
| `JD_GENERATING` | 正在生成JD | （等待AI完成） | → `JD_CONFIRMING` |
| `JD_CONFIRMING` | JD已生成，等待确认 | `确认` / `OK` | → `COLLECTING` |
| | | `修改: ...` | → `JD_GENERATING` |
| | | 超时(5min) | → `IDLE` |
| `COLLECTING` | 等待HR提交简历 | 发送简历文件/文本 | 保持 `COLLECTING`，resume_count+1 |
| | | `@机器人 筛选` | → `SCREENING`（需简历≥3） |
| | | `@机器人 进度` | 保持当前状态，推送进度 |
| `SCREENING` | 正在评分 | （等待AI评分完成） | → `SCREENING_RESULT` |
| `SCREENING_RESULT` | 报告已推送 | `@机器人 筛选` | → `SCREENING`（重新评分） |
| | | `@机器人 帮我招新岗位` | → `JD_GENERATING`（重置） |
| | | 超时(5min) | → `IDLE` |

**全局重置规则**：任何状态 + HR发送"帮我招一个XX" → 立即重置状态机，开始新任务。旧岗位数据保留在数据库中，可在之后的对话中通过岗位名称再次关联。

**技术实现**：
- Redis `SET key value EX 300` 每次状态更新时刷新 TTL
- 状态切换使用 Lua 脚本保证原子性
- 企微回调服务在接收消息时，先查 Redis 确定当前状态，再路由到对应 Dify 工作流节点

#### 7.2.2 多条简历同时提交（详细方案）

**问题**：HR 可能在数秒内拖入 5 份 PDF 简历到企微群。企微会将每份文件作为独立消息推送给回调服务。每份简历解析耗时约 30-60 秒。如何保证处理有序、不丢文件、且提供清晰的进度反馈？

**方案：批次窗口 + Redis队列 + 汇总反馈**

**步骤1 — 入队（5s窗口合并）**

```
HR拖入 5 份PDF → 企微逐条推送回调
                   ↓
回调服务接收第1份 → 写入 Redis队列 "batch:{group_id}:{timestamp_5s_bucket}"
                   ↓
第2-5份在5s内到达 → 追加到同一批次队列
                   ↓
5s后批次关闭       → 触发 Consumer 开始逐份处理
```

**步骤2 — 逐份处理 + 进度反馈**

Consumer 从队列 FIFO 取出文件：

```
处理 resume_1.pdf (1/5):
  → 解析完成 → 结构化存储
  → 推送进度消息到群:
    "📄 简历解析中 (1/5)
     resume_1.pdf: 张三 · 芯片验证工程师
     剩余4份处理中，完成后推送汇总..."

处理 resume_2.pdf (2/5):
  → 解析完成 → 存储
  → 更新上一条进度消息为 (2/5):
     "📄 简历解析中 (2/5)
      resume_2.pdf: 李四 · 嵌入式开发..."

...逐一处理至 5/5 ...

处理 resume_5.pdf (5/5):
  → 解析完成 → 存储
  → 推送汇总报告:
     "✅ 5份简历解析完成
      成功: 5 | 失败: 0
      关联岗位: 嵌入式软件工程师
      💡 回复「开始筛选」进行AI评分"
```

**进度消息策略**：
- 每完成一份，更新同一条企微消息（通过 msg_id 覆盖更新），而非发送 5 条独立消息
- 如果企微不支持消息更新，则只在最后一份完成时发送汇总，中间不推送进度
- 如果处理耗时超过 30 秒，推送一条"处理中，请稍候..."的安慰消息

**步骤3 — 异常处理**

| 异常场景 | 处理策略 |
|---------|---------|
| 某份简历解析失败 | 跳过，记录原因，汇总报告中标注 `⚠️ resume_3.pdf 解析失败：非标准格式` |
| 批次处理中途服务重启 | Redis持久化队列，重启后从断点继续 |
| HR在处理中又提交新文件 | 创建新批次，队列追加；完成当前批次后继续处理 |
| 单份简历超过3MB | 企微文件限制20MB，简历通常 < 3MB，超限时提示压缩后重发 |

**技术实现**：
- 使用 Redis List (`LPUSH`/`BRPOP`) 作为批次队列
- 批次 key: `batch:{group_id}:{unix_ts_div_5}`（每5秒一个桶）
- Consumer 使用 `BRPOP` 阻塞读取，无任务时休眠
- 进度消息 ID 记录在 `batch:{group_id}:{ts}:progress_msg_id` 中，用于后续更新

#### 7.2.3 群内多HR协作

**问题**：同一个企微群中可能有多个HR，他们都可能@机器人操作同一岗位。

**方案**：MVP阶段限定为「单群单活跃岗位」——每个群同时只有一个活跃的招聘流程。多人操作时，状态机共享，最后操作生效。操作日志记录每条操作人和时间，避免责任不清。

Q3可升级为「角色权限」——区分岗位创建者（Owner）和协作者（Contributor），仅Owner可触发筛选和修改JD。

| **候选人数据授权** | 《个人信息保护法》第13条：处理个人信息需取得个人同意。AI自动解析简历、评分需候选人明确授权 | 简历投递时增加授权勾选："我同意该公司使用AI辅助工具对我的简历进行自动化处理"。人才库归档需二次确认 | 🔴 法律硬需求 |
| **AI决策透明度** | 候选人可能对AI筛选结果提出异议，《个人信息保护法》第24条要求自动化决策的透明性 | 提供筛选依据说明（每个维度的得分理由）+ 人工复核机制 + 申诉通道 | 🔴 法律硬需求 |
| **数据本地化** | 简历数据在哪些服务器处理？是否出境？ | 全部数据和LLM调用限定在国内服务器，避免数据出境风险 | 🔴 需确认 |
| **数据留存期限** | 候选人简历应保留多久？ | 建议：活跃岗位存续期 + 6个月，之后脱敏或删除；人才库留存需单独授权 | 🟡 需HR+法务确认 |
| **跨平台数据获取** | 从项目管理平台/人事系统获取数据需确认数据权限范围 | IT需确认API返回的字段权限 + 最小够用原则 | 🟡 Q3前确认 |

### 7.4 数据问题

| 问题 | 风险 | 建议方案 | 状态 |
|------|------|---------|:------:|
| **简历格式多样性** | PDF排版差异巨大（双栏/表格/图文混排），解析准确率不稳定 | MVP设定80%准确率目标 + 低置信度字段标注⚠️ + 提供"纠错"交互 | 🟡 需真实样本测试 |
| **评分模型冷启动** | 无历史面试结果数据，无法校准评分权重 | MVP用规则+LLM混合模型 + 标注5个维度的初始权重 + Q3收集数据迭代 | 🟡 先跑通再优化 |
| **岗位群隔离** | 不同群聊的招聘数据需要隔离，但技术上是同一数据库 | 通过群聊ID + CorpID进行数据隔离 | 🟡 需实现 |
| **JD模板积累** | 不同岗位类型需要不同的JD模板，初始可能不完善 | MVP用HR提供的5份历史JD作为Few-shot + 逐步积累模板库 | 🟡 需HR提供 |
